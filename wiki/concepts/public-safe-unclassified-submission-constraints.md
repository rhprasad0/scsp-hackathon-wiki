# Public-safe unclassified submission constraints

Public-safe unclassified submission constraints are the submission rules that require hackathon projects to use only public, unclassified, or otherwise shareable data and to avoid controlled or restricted content. In the GenAI.mil challenge materials, this includes an explicit warning to mind ITAR and EAR, and to ensure any preexisting code has a determined export classification before use. ^[2026-04-25-genai-mil-challenge-prompt-and-rubric.md]

These constraints shape both the data and the product. Teams are instructed that public submissions only are allowed, that no classified materials may be included, and that the work should stay within an unclassified, public-safe posture. The challenge also frames promising solutions around approved public DoD text, public APIs, and back-office workflows rather than tactical or kinetic use cases. ^[2026-04-25-genai-mil-challenge-prompt-and-rubric.md]

The submission model also requires the final repository to remain public throughout the hackathon and beyond, with a public GitHub repo and a README that explains the build, datasets or APIs used, and how to run the project. This means the constraint is not only about content selection, but also about the visibility and publication status of the deliverable itself. ^[2026-04-25-genai-mil-challenge-prompt-and-rubric.md]

In practice, these rules reinforce a [[public-safe]] and [[unclassified]] workflow: teams can build around public sources such as Army publications, Field Manuals, and other organizer-pointed public datasets, but they should not rely on controlled, restricted, or classified inputs. The challenge materials also tie this to the broader administrative-trenches framing, which favors reviewable, back-office systems over sensitive operational systems. ^[2026-04-25-genai-mil-challenge-prompt-and-rubric.md]
